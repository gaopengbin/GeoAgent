## 2. 核心数据流

### 2.1 一次完整对话的数据流

```
用户输入文本 → 前端ChatPanel
    │
    ▼ POST /api/chat (SSE)
    │
FastAPI接收 → 创建/恢复Session
    │
    ▼
LangGraph Agent (ReAct 循环)
    │ System Prompt + 用户消息 + 动态工具组 + SessionDataContext 数据清单
    │
    ▼ 循环：每次只决定一个工具调用
    │
    ├─▶ Step 1: geocoding("北京五环")
    │   └→ GisToolAdapter → 自建工具(天地图API) → ref_001
    │
    ├─▶ Step 2: fetch_osm_data("地铁站", area="北京")
    │   └→ GisToolAdapter → 自建工具(Overpass API) → ref_002
    │
    ├─▶ Step 3: buffer(ref_002, 500m)
    │   └→ GisToolAdapter → resolve ref_002 → MCP Client
    │     └→ gis-mcp Server: buffer() → store result → ref_003
    │
    ├─▶ Step 4: spatial_query("intersects", ref_003, ref_004)
    │   └→ GisToolAdapter → resolve refs → MCP Client
    │     └→ gis-mcp Server: sjoin_gpd() → store result → ref_005
    │
    └─▶ Step 5: generate_map_command("addGeoJsonLayer", ref_005)
        └→ 自建工具 → SSE 推送前端
    │
    ▼ Agent 判断：任务完成
    │
生成最终回复（聚合所有步骤结果）
    │
    ├─ 文本回复（流式SSE）: "分析完成，共找到XXX个小区..."
    ├─ 地图指令（JSON）: {action: "addHeatmap", data: {...}, flyTo: {...}}
    └─ 图表数据（JSON）: {type: "bar", data: {...}}
    │
    ▼ SSE Event Stream
    │
前端接收：
    ├─ ChatPanel: 逐字显示AI回复
    ├─ MapView: 执行地图指令 → 渲染热力图、飞行定位
    └─ ResultPanel: 渲染统计图表
```

### 3.2 Agent State定义

from typing import TypedDict, Annotated, Sequence
from langchain_core.messages import BaseMessage
from langgraph.graph.message import add_messages

class GeoAgentState(TypedDict):

### 3.4 大模型选型与分层路由

GeoAgent 采用**多模型可配置 + 任务分层路由**策略，所有模型均通过 OpenAI 兼容接口接入，配置即切换。

#### 模型层级

| 层级 | 用途 | 触发条件 | 配置项 |
|------|------|---------|--------|
| **快速层** | 简单指令（flyTo/底图切换/闲聊） | 意图识别为单步操作 | `LLM_FAST_MODEL` |
| **主力层** | 常规对话和工具调用 | 默认 | `LLM_MODEL` |
| **增强层** | 复杂多步分析（工具链>5步） | 任务规划器判定 | `LLM_ENHANCED_MODEL` |
| **推理层** | 多约束决策（选址/对比分析） | 用户显式选择或自动升级 | `LLM_REASONING_MODEL` |

#### 路由逻辑

```python
def select_model(intent: str, estimated_steps: int, user_override: str = None) -> str:
    """根据任务复杂度自动选择模型层级，用户可手动覆盖"""
    if user_override:
        return user_override  # 前端模型选择器覆盖

    if intent in ("flyTo", "setBasemap", "toggleLayer", "chitchat"):
        return settings.llm_fast_model
    elif estimated_steps > 5 or intent == "multi_step_analysis":
        return settings.llm_enhanced_model
    elif intent in ("site_selection", "comparison", "constraint_optimization"):
        return settings.llm_reasoning_model
    else:
        return settings.llm_model  # 主力层
```

### 3.5 MCP 工具架构与动态工具组

GeoAgent 采用 **MCP-first** 工具架构：GIS 空间分析能力由 [gis-mcp](https://github.com/mahdin75/gis-mcp)（103个原子工具）提供，通过 MCP Client 调用；业务工具（geocoding/数据获取/地图指令）自建为 @tool。中间通过 **GisToolAdapter** 桥接 data_ref_id 与原始数据。

#### 为什么不直接暴露 103 个工具给 Agent？

行业研究（[RAG-MCP 论文](https://arxiv.org/html/2505.03275v1)、[GitHub MCP Server](https://github.com/github/github-mcp-server/issues/275)）证实：

| 工具数 | LLM选择准确率 | 来源 |
|--------|-------------|------|
| <30 | >90% | RAG-MCP 压力测试 |
| 31-70 | 间歇性下降 | 语义重叠增多 |
| >100 | 显著退化 | Prompt Bloat 效应 |

#### 动态工具组（Dynamic Toolsets）

```python
@tool
def discover_gis_tools(category: str) -> dict:
    """按需启用GIS工具组。

    可用类别：
    - spatial_analysis: 缓冲区、空间查询、叠加分析、裁剪
    - measurement: 距离计算、面积计算、字段统计
    - geometry: 质心、凸包、简化、泰森多边形
    - coordinate: 坐标转换、UTM分区
    - visualization: 热力图、填色图、统计图表

    调用时机：当核心工具不足以完成任务时，用此工具启用对应分组。
    示例：用户要做缓冲区分析 → discover_gis_tools("spatial_analysis")
    """
    tools = TOOLSETS.get(category, [])
    # 动态注册到当前 Agent 的工具列表
    return {"enabled_tools": tools, "count": len(tools)}
```

#### GisToolAdapter（data_ref_id 桥接层）

```python
class GisToolAdapter:
    """gis-mcp 与 SessionDataContext 之间的桥接层"""

    def __init__(self, session_ctx: SessionDataContext, mcp_session):
        self.ctx = session_ctx
        self.mcp = mcp_session  # MCP Client Session (连接 gis-mcp)

    async def call_gis_tool(
        self, gis_tool_name: str,
        ref_params: dict[str, str],    # {"geometry": "ref_001"} 需要解析的引用参数
        extra_params: dict,            # {"distance": 500} 直接传递参数
    ) -> dict:
        # 1. data_ref_id → 原始数据（WKT/GeoJSON）
        resolved = {}
        for param, ref_id in ref_params.items():
            geojson = self.ctx.get(ref_id)
            if not geojson:
                available = [r["data_ref_id"] for r in self.ctx.list_refs()]
                raise ValueError(f"data_ref_id '{ref_id}' 不存在。可用: {available}")
            resolved[param] = self._to_wkt(geojson)

        # 2. 调用 gis-mcp 工具
        result = await self.mcp.call_tool(gis_tool_name, {**resolved, **extra_params})

        # 3. 结果转 GeoJSON → 存入 SessionDataContext
        result_geojson = self._parse_mcp_result(result)
        new_ref_id = self.ctx.register(
            result_geojson,
            source=f"gis-mcp:{gis_tool_name}({', '.join(ref_params.values())})"
        )

        return {
            "data_ref_id": new_ref_id,
            **self.ctx.summary(new_ref_id),
        }
```

#### @tool wrapper 示例

每个 Agent 可见的工具是一个薄包装，调用 GisToolAdapter 完成实际工作：

```python
@tool
def buffer_analysis(data_ref_id: str, distance: float, unit: str = "m") -> dict:
    """对指定图层创建缓冲区。

    使用场景：当用户说"XX范围内"、"周围XX米"时使用。
    需要先用 discover_gis_tools("spatial_analysis") 启用此工具。

    示例：
    - "地铁站500米范围内" → buffer_analysis(data_ref_id="ref_001", distance=500)
    - "公园周围2公里" → buffer_analysis(data_ref_id="ref_parks", distance=2, unit="km")
    """
    adapter = get_gis_adapter()
    d = distance * 1000 if unit == "km" else distance
    return await adapter.call_gis_tool(
        "buffer",
        ref_params={"geometry": data_ref_id},
        extra_params={"distance": d},
    )
```

> **设计原则**：Agent 看到的是业务语义工具（~15个），实际执行通过 GisToolAdapter 代理到 gis-mcp 的 103个原子工具。动态工具组确保每轮 Agent 上下文中工具数 ≤10，保持 >90% 的选择准确率。

### 3.6 Agent 可靠性工程

GIS Agent 的核心挑战是 LLM tool calling 的可靠性。采用 **5层防御体系** 保障工具调用链路的稳定性。

#### 第1层：工具设计原则

| 原则 | 说明 | 示例 |
|------|------|------|
| **单一职责** | 每个工具只做一件事 | `buffer_analysis` 只做缓冲区，不合并空间查询 |
| **参数≤5个** | 参数越少LLM越不容易出错 | 必填参数尽量≤3个 |
| **data_ref_id 传递** | 工具间通过引用ID传数据 | 不传完整GeoJSON，避免上下文爆炸 |
| **描述写使用场景** | 不只描述功能，还写触发条件 | "当用户说XX范围内时使用此工具" |
| **描述含Few-shot** | 工具描述中嵌入调用示例 | 减少LLM对参数格式的猜测 |
| **参数用enum约束** | 能用enum的绝不用str | `unit: Literal["m", "km"]` |

```python
# 工具设计示范
@tool
def buffer_analysis(
    data_ref_id: str,     # 输入数据引用ID（来自上一步工具输出）
    distance: float,      # 缓冲距离，单位：米
    unit: str = "m",      # "m" | "km"
) -> dict:
    """对指定图层创建缓冲区。

    使用场景：当用户说"XX范围内"、"周围XX米"、"XX公里以内"时使用。
    输入：data_ref_id 是之前工具返回的数据引用ID。
    输出：返回缓冲区GeoJSON的新data_ref_id。

    示例调用：
    - "地铁站500米范围内" → buffer_analysis(data_ref_id="ref_subway", distance=500)
    - "公园周围2公里" → buffer_analysis(data_ref_id="ref_parks", distance=2, unit="km")
    """
```

#### 第2层：ReAct 逐步执行

采用 LangGraph 的 ReAct 循环，**每次只决定并执行一个工具调用**，而非一次规划全部步骤：

```
循环：
1. LLM 观察当前状态（用户意图 + 已完成步骤 + 已有数据清单）
2. LLM 决定下一个工具调用（只调一个）
3. 执行工具，将结果摘要加入上下文
4. 将工具执行步骤实时推送给前端（SSE tool_call / tool_result 事件）
5. LLM 判断：是否还需要继续？
   - 是 → 回到步骤1
   - 否 → 生成最终回复
```

> **关键**：不让LLM一次规划N步（误差累积），而是每步拿到真实结果后再决定下一步。

#### 第3层：结构化上下文注入

每次调用LLM时，动态注入当前 `SessionDataContext` 的数据清单：

```python
def build_context_prompt(session_ctx: SessionDataContext) -> str:
    """注入当前会话已有的数据引用清单"""
    refs = session_ctx.list_refs()
    if not refs:
        return "当前会话中没有已加载的数据。"

    lines = ["当前会话中已有以下数据可供引用："]
    for ref in refs:
        lines.append(
            f"- data_ref_id=\"{ref['id']}\" | "
            f"名称: {ref['name']} | "
            f"类型: {ref['geometry_type']} | "
            f"要素数: {ref['feature_count']} | "
            f"来源: {ref['source']}"
        )
    lines.append("\n调用工具时，使用 data_ref_id 引用上述数据，不要编造不存在的ID。")
    return "\n".join(lines)
```

#### 第4层：参数验证 + 错误反馈

工具执行前验证参数合法性，失败时将错误信息返回给LLM进行自我修正（而非直接崩溃）：

```python
def validate_tool_params(tool_name, args, session_ctx):
    """执行前校验"""
    # 1. data_ref_id 存在性检查
    if "data_ref_id" in args:
        if not session_ctx.get(args["data_ref_id"]):
            available = [r["id"] for r in session_ctx.list_refs()]
            return f"data_ref_id '{args['data_ref_id']}' 不存在。可用: {available}"

    # 2. 必填参数检查
    schema = TOOL_SCHEMAS.get(tool_name)
    if schema:
        for field in schema.get("required", []):
            if field not in args:
                return f"缺少必填参数: {field}"

    return None  # 校验通过
```

> 错误信息返回给LLM后，LLM会在下一轮自动修正参数重新调用，而非终止整个对话。

## 7. 数据存储分层策略

数据不全部入库，按生命周期、数据量、访问模式分为三级：

### 7.1 Level 0：纯内存（用完即弃）

适用于体积极小（<1KB）、一次性消费、无后续引用的数据：

| 数据类型 | 来源工具 | 理由 |
|----------|---------|------|
| geocoding坐标 | `geocoding` | 单个坐标点，下游工具直接消费 |
| 距离/面积/统计值 | `distance_calculation`, `spatial_statistics` | 纯数值，无几何 |
| 地图指令JSON | `generate_map_command` | 推送后记录在message表即可 |
| 图表数据 | 各分析工具 | 纯数值数组 |
| 报告Markdown | `generate_report` | 存在message表 |

### 7.2 Level 1：会话级缓存（data_ref 引用）

适用于中等数据量（1KB-10MB）、同会话可能被多次引用的空间数据。工具间通过 `data_ref_id` 引用传递，避免全量GeoJSON拷贝。

```python
# 会话数据上下文（内存字典，MVP阶段；后续可迁移到Redis）
session_data_context = {
    "session_id": "uuid-xxx",
    "data_refs": {
        "ref_subway_stations": {
            "data": <GeoJSON FeatureCollection>,
            "source": "fetch_osm_data",
            "created_at": "2024-01-01T00:00:00Z",
            "feature_count": 347,
            "geometry_type": "Point",
            "size_bytes": 125000
        },
        "ref_subway_buffer_500m": {
            "data": <GeoJSON FeatureCollection>,
            "source": "buffer_analysis",
            "created_at": "2024-01-01T00:00:05Z",
            "feature_count": 347,
            "geometry_type": "Polygon",
            "size_bytes": 890000
        }
    }
}
```

适用数据：

| 数据类型 | 来源工具 | 理由 |
|----------|---------|------|
| OSM/天地图查询结果（<5000要素） | `fetch_osm_data`, `fetch_tianditu` | 同会话可能多次引用 |
| 用户上传小文件（<1MB） | `load_uploaded_file` | 全量内存足够 |
| 分析中间结果 | `buffer_analysis`, `spatial_query`, `overlay_analysis` | "把缓冲区改成1000米"需引用之前数据 |
| 热力图/聚类结果 | `kernel_density`, `cluster_analysis` | 中等量，用于前端渲染 |

### 7.3 Level 2：PostGIS 持久化

适用于大数据量（>10MB）、需要空间索引、需要跨会话持久化的数据：

| 数据类型 | 入库理由 |
|----------|---------|
| 用户上传大文件（>1MB） | 内存放不下，且需空间索引加速 |
| 需要 ST_Within/ST_Intersects 的分析 | GIST索引比内存Shapely快10-100倍 |
| 最终展示图层 | 展示在图层面板，需持久化 |
| 会话/消息/图层元数据 | 业务数据，必须持久化 |
| 跨会话复用的数据 | 用户下次访问仍需使用 |

### 7.4 自动分级决策

```
数据产生 → 是否含几何? ─ NO ──→ Level 0 (内存)
                │ YES
                ├─ 数据量 > 10MB? ─ YES ──→ Level 2 (PostGIS)
                │ NO
                ├─ 需要空间索引? ── YES ──→ Level 2 (PostGIS)
                │ NO
                ├─ 需要跨会话? ─── YES ──→ Level 2 (PostGIS)
                │ NO
                └─ 同会话可能复用? ─ YES → Level 1 (会话缓存)
                                    NO ──→ Level 0 (内存)
```

### 7.5 Level 1 → Level 2 自动升级

当会话缓存中的数据满足以下条件时，自动升级到 PostGIS：
- 单个 data_ref 超过 10MB
- 会话总缓存超过 100MB
- 用户主动要求保存图层（"帮我保存这个分析结果"）
- 数据被标记为最终展示图层
